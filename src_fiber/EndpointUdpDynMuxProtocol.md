# Dynamic UDP Multiplexing Protocol (2-Byte Channel ID)

This document defines the protocol for dynamic Peer ID negotiation, keepalive-based session lifecycle management, address migration, and error pings under the `great-hole` UDP multiplexing scheme.

---

## 1. General Framing & Multiplexing

All multiplexed packets sent over the UDP socket start with a **2-byte Channel ID** in big-endian (network byte order):
- **Channel ID `0x0000`**: Reserved for the Control & Negotiation Channel.
- **Channel ID `0x0001` to `0xFFFF`**: Data Channels. The rest of the packet contains pure, raw payload data.

Because the data channels have a 2-byte prefix and do not have any internal headers or parsing overhead, they maintain 100% zero-overhead data transfer.

---

## 2. Control Messages (Channel ID `0x0000`)

For any packet starting with `0x0000`, the 3rd byte (offset `2`) specifies the **Message Type**.

### `0x01`: `CLIENT_REQ_ID`
*   **Direction**: Client $\rightarrow$ Server
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x01] [4-byte: Client Cookie]`
*   **Client Cookie**: A cryptographically random 32-bit integer generated by the client to uniquely identify this negotiation session.
*   **Purpose**: Client requests a dynamic peer ID from the server.

### `0x02`: `SERVER_ASSIGN_ID`
*   **Direction**: Server $\rightarrow$ Client
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x02] [4-byte: Client Cookie] [2-byte: Assigned Peer ID]`
*   **Assigned Peer ID**: Network byte order 16-bit integer (1-65535). If `0x0000`, it indicates that allocation failed (e.g., server channel pool is full).
*   **Purpose**: Assigns the negotiated peer ID to the client.

### `0x03`: `SERVER_KEEPALIVE`
*   **Direction**: Server $\rightarrow$ Client
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x03] [2-byte: Peer ID]`
*   **Purpose**: Periodically sent by the server to verify the client is still active.

### `0x04`: `CLIENT_KEEPALIVE_ACK`
*   **Direction**: Client $\rightarrow$ Server
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x04] [2-byte: Peer ID]`
*   **Purpose**: Sent by the client to acknowledge a keepalive ping.

### `0x05`: `SERVER_ID_CLOSED`
*   **Direction**: Server $\rightarrow$ Client
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x05] [2-byte: Peer ID]`
*   **Purpose**: Notifies the client that its session ID was closed or released.

### `0x06`: `SERVER_ADDR_MISMATCH`
*   **Direction**: Server $\rightarrow$ Client
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x06] [2-byte: Peer ID]`
*   **Rate Limiting**: Shares the unified 1-second per peer endpoint error rate-limit bucket.
*   **Purpose**: Sent by the server when it receives a data packet for an active ID but from an unrecognized source IP/port, triggering client address migration.

### `0x07`: `CLIENT_ADDR_MIGRATE`
*   **Direction**: Client $\rightarrow$ Server
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x07] [2-byte: Peer ID] [4-byte: Original Client Cookie]`
*   **Original Client Cookie**: The cryptographically secure cookie used during initial negotiation.
*   **Purpose**: Sent by the client from its new source address to request a session IP/port migration.

### `0x08`: `SERVER_MIGRATE_ACK`
*   **Direction**: Server $\rightarrow$ Client
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x08] [2-byte: Peer ID]`
*   **Purpose**: Acknowledges a successful session IP/port migration.

### `0x09`: `SERVER_INVALID_ID`
*   **Direction**: Server $\rightarrow$ Client
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x09] [2-byte: Unknown Peer ID]`
*   **Rate Limiting**: Shares the unified 1-second per peer endpoint error rate-limit bucket.
*   **Purpose**: Sent by the server when it receives a packet with an unrecognized peer ID.

### `0x0A`: `SERVER_COOKIE_MISMATCH`
*   **Direction**: Server $\rightarrow$ Client
*   **Format**: `[2-bytes: 0x0000] [1-byte: 0x0A] [2-byte: Peer ID]`
*   **Rate Limiting**: Shares the unified 1-second per peer endpoint error rate-limit bucket.
*   **Purpose**: Sent by the server when a migration request (`CLIENT_ADDR_MIGRATE`) is received with an invalid cookie.

## 3. Client State Machine

```
   [none] -- (async_start) --> [negotiating] <---------+
                                    |                  | (keepalive timeout OR
                                    | (assigned ID)    |  SERVER_INVALID_ID OR
                                    v                  |  SERVER_COOKIE_MISMATCH)
   [migrating] <-- (mismatch) -- [running] ------------+
        |                            ^
        +---- (migrate ACK) ---------+
```

- **Negotiation**: Retries `CLIENT_REQ_ID` every 1 second with a random cookie until `SERVER_ASSIGN_ID` matches.
- **Queueing**: During `negotiating` or `migrating` states, all outgoing user packets passed to `async_write` are queued in `write_queue`. When state goes to `running`, `write_queue` is flushed sequentially.
- **Continuous Receive Loop**: The client runs a continuous background socket read loop to receive control/keepalive packets independently of user data consumption. Data packets are queued in `incoming_data_queue` (capped at 1000 packets) if no read handler is currently registered.
- **Migration**: When receiving `SERVER_ADDR_MISMATCH`, the client goes to `migrating`, queues data packets, and retries `CLIENT_ADDR_MIGRATE` (containing its original cookie) until `SERVER_MIGRATE_ACK` is received. If a `SERVER_COOKIE_MISMATCH` is received, it triggers a fresh negotiation.
- **Keepalive**: Client updates a `last_keepalive_received` timestamp on keepalive messages. If more than 15 seconds pass without keepalives, it triggers re-negotiation.

---

## 4. Server Logic

- **ID Pool**: Server registers channel IDs via `create_channel(id)` in a `std::set<uint16_t> created_channels`. A free ID is any ID in `created_channels` not present in `peers` (active sessions).
- **Session Mapping**: Server maps active peer IDs to endpoints in `peers` and original cookies in `cookies`.
- **Keepalive Timer (Every 5 seconds)**: Server pings all active endpoints. If a client fails to ACK for 3 consecutive ticks (15 seconds), the ID is released, resources cleared, and pending channels are closed.
- **Migration & Verification**:
  - Validates source IP & Port of all data/ACK packets. Mismatches trigger a rate-limited `SERVER_ADDR_MISMATCH`.
  - When receiving `CLIENT_ADDR_MIGRATE`, checks if the cookie matches the original. If valid, updates `peers[id]` to the new address and replies with `SERVER_MIGRATE_ACK`. If invalid, replies with `SERVER_COOKIE_MISMATCH`.
- **Rate Limiting Guardrails**:
  - All control error messages (`SERVER_ADDR_MISMATCH`, `SERVER_INVALID_ID`, and `SERVER_COOKIE_MISMATCH`) sent to a specific client destination address are Collectively Rate Limited. The server maintains a unified error timestamp map (`last_error_sent`) keyed by the client's destination UDP endpoint. If an error message was sent to that endpoint less than 1 second ago, the new error packet is dropped. This completely prevents network amplification and DoS attacks.
